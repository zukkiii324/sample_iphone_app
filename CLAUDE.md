このファイルは、AIアシスタント（Claude など）がこのリポジトリで作業する際に守るべきルールや、プロジェクトの全体像をまとめたものです。

---

## プロジェクト概要

リポジトリ名は `sample_iphone_app` ですが、**iOSアプリではありません**。
GitHub Pages で公開する**静的Webサイト**（HTML と CSS だけで作られたサイト）です。

ホームページと、カテゴリごとのサブページで構成されています。
JavaScript やサーバーサイドの仕組みは使っていません。

## CSS の設計ルール

### 共通スタイル（`assets/css/base.css`）

すべてのページで使うスタイルは `base.css` にまとめています。
色やフォントなどは「CSS カスタムプロパティ（CSS変数）」で管理しています。

よく使う変数の一覧：

| 変数名 | 役割 | 初期値の例 |
|--------|------|-----------|
| `--color-bg` | ページの背景色 | `#f3f6fb` |
| `--color-surface` | カードなどの背景色 | `#ffffff` |
| `--color-text` | 本文の文字色 | `#1f2937` |
| `--color-muted` | 補足テキストの色 | `#64748b` |
| `--color-accent` | カテゴリごとの強調色 | `#2563eb` |
| `--color-border` | 枠線の色 | `rgba(15,23,42,0.08)` |

### カテゴリごとのテーマカラー

各サブページの `styles.css` で `--color-accent` を上書きするだけで、ページ全体の色味を変えられます。

### CSSクラスの命名

ケバブケース（単語をハイフンでつなぐ書き方）を使います。

```
良い例: site-header, nav-links, category-page
悪い例: siteHeader, NavLinks, category_page
```

### 使用しているモダンCSS機能

- `color-mix()` — 色を混ぜて新しい色を作る関数
- CSS Grid / Flexbox — レイアウト用
- `backdrop-filter` — ヘッダーのすりガラス風エフェクト
- CSS カスタムプロパティ — 色やサイズの一元管理

## HTML の書き方ルール

### セマンティック HTML を使う

意味のあるタグを正しく使い分けてください。

| タグ | 使いどころ |
|------|-----------|
| `<header>` | ページ上部のヘッダー |
| `<nav>` | ナビゲーションメニュー |
| `<main>` | ページのメインコンテンツ |
| `<section>` | コンテンツのまとまり |
| `<article>` | 独立した記事やカード |
| `<footer>` | ページ下部のフッター |

### アクセシビリティ

- すべてのページに `lang="ja"` を設定する
- ナビゲーションには `aria-label` を付ける
- 見出しタグ（h1 → h2 → h3）は順番通りに使う
- SVG アイコンには `aria-hidden="true"` を付ける

## 新しいカテゴリページの追加手順

1. `pages/` の中に新しいフォルダを作る（例: `pages/health/`）
2. `index.html` を作成する
   - `../../assets/css/base.css` を読み込む
   - 既存のサブページを参考にしてレイアウトを合わせる
3. `styles.css` を作成し、`--color-accent` をそのカテゴリに合った色に設定する
4. トップページ（`index.html`）のナビゲーションに新カテゴリへのリンクを追加する

## 開発ルール（DEVELOPMENT_RULES.md より）

- **バックエンドは使わない** — データはすべてフロントエンド（HTML/CSS）内で管理する
- **DRY原則**（同じコードを繰り返さない） — 共通スタイルは `base.css` を再利用する
- **単一責務の原則** — 1つの関数やコンポーネントは1つの役割に絞る
- **ドキュメント更新** — コードを変更したら、関連するドキュメントも更新する
- **継承よりコンポジション** — 部品を組み合わせて作る設計を優先する
- **作業前にブランチを最新化** — コンフリクト（競合）を防ぐため、作業開始前に最新を取得する

---

## テスト方法

自動テストの仕組みはありません。変更後は以下の方法で確認してください。

1. **ブラウザで目視確認** — ページを開いて表示が崩れていないかチェック
2. **レスポンシブ確認** — 画面幅 920px 以下、720px 以下で表示を確認する
3. **複数ブラウザ確認** — Chrome、Safari、Firefox など主要ブラウザで動作チェック

---

## デプロイ（公開）方法

`main` ブランチにコードをプッシュすると、GitHub Pages が自動的にサイトを公開します。
特別なビルド手順やコマンドは不要です。
